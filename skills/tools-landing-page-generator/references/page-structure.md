## Required Page Contract

Create all eight content sections in this exact order:

1. HERO
2. Why Choose
3. Key Features
4. How to
5. Comparison
6. FAQ
7. Privacy and Data Security
8. Final CTA

The site header and footer do not count as content sections. Keep the tool placeholder inside HERO. Do not move it into a ninth section. Do not insert additional content sections or change this order unless the user explicitly changes the contract.

Read `writing-style.md` before drafting. Use the keyword research report and user supplied product context as inputs. Item counts below are required. Word ranges apply to final rendered English copy but have the lowest priority when requirements conflict.

Use this conflict order: factual accuracy and effective SEO, then natural and clear expression, then `writing-style.md`, then this structure guide, then suggested word lengths. If an unusually long target keyword makes a word range impractical, keep the keyword natural and accurate, allow the minimum necessary length exception, and record the measured length and reason in the QA report.

## Evidence Gates

Use facts from the user, the target site's official pages, official product documentation, or other reliable sources. Apply these gates before drafting and again before finalizing:

- Verify product features, workflow steps, input and output formats, technical limits, pricing, free access, registration rules, compatibility, speed, accuracy, and performance claims.
- Verify every privacy and data security statement, including storage, deletion, account requirements, protection methods, and third party sharing.
- Verify ratings, usage numbers, customer counts, awards, certifications, and superlatives.
- Support comparison claims with evidence gathered during research. Do not infer that a competitor lacks a feature merely because it was not found.
- Use `Not publicly stated` for an unresolved competitor table value. Do not use an unsupported cross, checkmark, ranking, or negative claim.
- Keep unresolved evidence notes in intermediate artifacts only. Ask the user for missing target product facts. Do not ship unsupported claims or internal markers in final HTML.

## 1. HERO

### Goal

Make the product and its practical value clear at first glance.

### Required Elements

1. H1
2. Description
3. Tool placeholder

### H1 Rules

- Use five to ten words.
- Include the exact target keyword naturally, preferably near the beginning.
- Incorporate a verified core value proposition and useful high frequency wording from the research report when natural.
- Do not include the brand name.
- Do not use dash punctuation.
- State what the tool is or what it helps the reader do. Avoid vague claims.

### Description Rules

- Use 20 to 25 words.
- State both the core function and the main benefit to the user.
- Use relevant high frequency terms naturally.
- Keep the language simple, direct, and evidence based.

### Tool Placeholder Rules

- Place a clearly identifiable tool integration placeholder inside HERO.
- Treat it as a reserved interface area, not a separate page section.
- Match its visual treatment to the extracted site style.
- Label it clearly in the HTML source so a developer can replace it later.
- Do not simulate uploads, generation, downloads, or results as working features unless the user supplied a real implementation.

## 2. Why Choose

### Goal

Explain why the target product may suit the reader's needs without using hype or unsupported superiority claims.

### Required Shape

- One section heading containing both the brand name and the target keyword
- Exactly four value points
- Each point: short heading plus a description of about 20 words

### Heading Rule

Use a neutral, specific heading such as:

> Why Use [Brand] for [Target Keyword]

Do not use `best choice`, `number one`, `leading`, or a similar superlative unless the user provides strong, publishable evidence. The brand name and target keyword remain required even when the example wording changes.

### Content Rules

- Base the four points on researched core value propositions and high value topics.
- Focus on practical differences such as ease of use, workflow, output, privacy, or fit for a verified use case.
- Do not claim that the product is faster, safer, more realistic, or better suited to a group without evidence.
- Give each point a distinct job and avoid repeating feature copy.

## 3. Key Features

### Goal

Connect real product capabilities with the people, situations, problems, and benefits they serve.

### Required Shape

- One section heading
- One introductory sentence of about 20 words
- Exactly six feature blocks
- Each feature: short heading, description of about 80 words, and one CTA button

### Feature Description Rules

Each feature description must explain:

1. What the feature is
2. How the reader uses it
3. Who it is for
4. A realistic use case
5. The problem it addresses
6. The practical benefit

Use the high value topics, user profile, and six typical use cases from the research report. Split an approximately 80 word description into short sentences or two brief paragraphs when needed to preserve readability. Do not repeat the same benefit across all six features.

Every feature and CTA must match verified product behavior. Do not invent a feature to fill the required count. Ask the user when fewer than six distinct supported capabilities can be established.

## 4. How to

### Goal

Show that the real product workflow is easy to understand and follow after the reader understands the product's key features.

### Required Shape

- Heading: `How to [Complete the Main Task]`
- Intro: one sentence of about 20 words
- Steps: three or four
- Each step: action led title plus a description of about 30 words

### Writing Rules

- Start every step title with a verb, such as `Upload a File`, `Choose a Style`, `Generate the Result`, or `Download the Result`.
- Explain the actual action the user takes.
- Explain the immediate result when useful.
- Keep the sequence clear and easy to follow.
- Verify the workflow against product evidence. If the real steps are unknown, ask the user before drafting this section.

## 5. Comparison

### Goal

Help the reader understand relevant differences without unsupported negative claims.

### Required Shape

- One neutral comparison heading
- One introductory sentence of about 20 words
- One comparison table
- Six product entries in total: the target product plus exactly five real competitors

### Heading Rule

Prefer a descriptive heading such as:

> How [Target Product] Compares for [Target Keyword]

This replaces salesy wording such as `What Makes Our Tool Different` while keeping the comparison purpose clear.

### Table Rules

- Select the five competitors from the target keyword research, not from guesswork.
- Choose dimensions from researched core value propositions and high value topics.
- Use the same dimensions and evaluation standard for every product.
- Cite or record supporting sources in the research artifact.
- Use precise factual wording instead of promotional labels.
- Write `Not publicly stated` when a competitor fact cannot be verified.
- Do not infer superiority from missing information.

## 6. FAQ

### Goal

Answer real questions, cover high value topics and long tail intent, and address useful objections directly.

### Required Shape

- Eight to ten questions
- One answer of 20 to 50 words for each question

### Answer Rules

- Answer the question immediately. Do not use a long setup.
- For a `How to` question, give a clear step by step answer.
- For a `What is` question, begin with `[Term] is...`.
- Cover a useful mix of usage, quality, price or free access, privacy, compatibility, limits, and comparison topics when evidence is available.
- Base questions on the research report's FAQ candidates, high value topics, long tail terms, semantically related terms, and forum discussions.
- Do not make up product facts to answer a plausible question. Ask the user or choose another evidenced question.

## 7. Privacy and Data Security

### Goal

Reduce uncertainty with concrete, verified information about how the product handles user data.

### Required Shape

- One section heading
- One introductory sentence of no more than 20 words
- Exactly four privacy or security points
- Each point: short heading plus no more than 20 words of description

### Topic Guidance

Use verified facts about:

- Whether user data or uploaded files are stored
- Whether and when files are deleted
- Whether registration is required
- Whether data is shared with third parties
- How data is protected

Privacy language is a strict evidence gate. Do not turn a missing policy into a promise. If four distinct publishable facts cannot be verified, ask the user before finalizing the page.

## 8. Final CTA

### Goal

Invite the reader to take the next real action in a clear, low pressure way.

### Required Shape

- One sentence heading
- One sentence description
- Optional button label, such as `Try Now`, `Start Free`, or `Upload Your File`

### Writing Rules

- Begin the heading or button with a direct action when natural, such as `Try`, `Start`, or `Upload`.
- Reinforce a verified core value proposition.
- Keep the request simple, specific, and non salesy.
- Use `free`, `fast`, or a result claim only when evidence supports it.
- Point the CTA to a real destination when the user provides one. Otherwise use a clearly marked implementation placeholder in the HTML source.

## Final Structure Check

Before approval, verify all of the following:

- All eight required sections exist once and in the required order.
- The tool placeholder is inside HERO.
- HERO has one H1 of five to ten words and one description of 20 to 25 words, unless a documented long keyword exception is necessary.
- Why Choose has four value points and a neutral heading containing the brand and target keyword.
- Key Features has six supported feature blocks and six CTA buttons.
- How to has three or four verified steps and appears immediately after Key Features.
- Comparison includes the target product and five real competitors.
- FAQ has eight to ten questions with answers of 20 to 50 words.
- Privacy and Data Security has four verified points.
- Final CTA has a one sentence heading and one sentence description.
- All copy follows `writing-style.md`, and no evidence gated claim remains unresolved.
- Every word length exception states the measured length and reason in the QA report.
